# 제품 요구 사항 정의서 (PRD: Product Requirements Document)
## 프로젝트명: 다했니? 쿠키 교환소 (Dahaenni Cookie Exchange Shop)

본 문서는 '다했니 Open API' 모의 연동을 활용하여, 학생들이 모은 쿠키 10개를 학급 상점 리워드 상품으로 실시간 교환하고 선생님이 이를 승인할 수 있도록 설계한 풀스택 애플리케이션의 제품 요구 사항 정의서입니다.

---

### 1. 프로젝트 개요 (Overview)
- **목적**: '다했니(교사용)' 및 '다했어요(학생용)' 플랫폼의 사용자 경험(UX)을 완벽히 재현하고, 쿠키 기반의 보상 체계를 웹 게임화(Gamification)하여 교사와 학생 간의 학급 관리를 더욱 즐겁고 직관적으로 개선합니다.
- **주요 기능**:
  - 교사용: 1반부터 10반까지 반(Class)별 보드 제공 및 실시간 쿠키 교환 요청 시 반 탭 및 학생 이름의 '시각적 색상 변경 경고' 알림 제공.
  - 학생용: 4자리 '다했어요 코드' 로그인, 쿠키 저금통 확인, 모달창 2차 확인 유도("쿠키 10개를 사용하시겠습니까?"), 교환 요청 및 자동 차감 확인.
- **실시간 통신 아키텍처**: Node.js/Express 백엔드를 통해 단일 상태 데이터를 소유 및 자동 실시간 피드백(폴링/업데이트)을 동기화하여 다인원 데모 플레이가 원활하도록 구현됩니다.

---

### 2. 사용자 타입 및 핵심 유저 스토리 (User Personas & Stories)

#### A. 학생 (다했어요 사용자)
- **로그인**: "나는 내 고유의 4자리 '다했어요 코드'를 입력하여 내 정보(이름, 쿠키 수, 소속 반)를 확인하고 싶다."
- **실시간 교환 신청**: "나는 모은 쿠키가 10개 이상일 때 상점에서 '급식 1등권'이나 '하루 자리 자유권'을 신청하고 싶다."
- **신중한 선택**: "실수로 버튼을 누르는 일을 방지하기 위해, 신청 전 '쿠키 10개를 사용하여 교환하시겠습니까?'라는 확인 경고창(확인과 돌아가기 버튼)을 보고 신중히 결정하고 싶다."
- **실시간 수령 확인**: "선생님이 승인하는 순간, 내 저금통에서 쿠키 10개가 뿅 하고 날아가며 교환 완료 애니메이션과 차감 소식을 즉각 확인하고 싶다."

#### B. 선생님 (다했니 관리자)
- **전체 학급 모니터링**: "나는 다했니 플랫폼 본래의 레이아웃처럼 1반부터 10반까지 탭을 클릭하며 학생들의 실시간 쿠키 적립 현황을 한눈에 보고 싶다."
- **실시간 요청 자동 감지 (비주얼 알림)**: "어느 반에서 교환 요청이 들어오든, 해당 반 탭에 빛 효과가 울리고, 해당 학생 카드가 특수 테두리 및 색상(감동적인 주황색/골드 그라데이션)으로 변경되어 즉각 확인하고 싶다."
- **교환 승인 완료**: "대시보드에서 '상점 교환 완료' 단일 버튼을 클릭하면 학생에게 완료 푸시가 가고 자동으로 쿠키 10개가 편리하게 차감 처리되길 원한다."
- **테스트용 수동 도구**: "학생들이 쿠키를 모으는 과정을 직접 데모로 시연하기 어려울 수 있으니, 선생님 보드에서 각 학생별로 쿠키를 `+1` / `-1` 증감할 수 있는 관리용 버튼이 유기적으로 내장되어 있으면 좋겠다."

### 3. 핵심 기능 상세 사양 (Functional Specifications)

#### 3.1 학생용 (다했어요) 기능
- **로그인 월 (Gate)**: 
  - 4자리 숫자 비밀코드 입력 필드와 큼직한 가상 키패드.
  - 테스트용 추천 코드 목록 제공 (각 반 대표 학생 코드 예: `1001`부터 `1010`까지).
- **쿠키 저금통 대시보드**:
  - 현재 보유한 쿠키 개수 표시.
  - 다채롭고 직관적인 아날로그 감성의 쿠키 그래픽.
- **상점 교환 리워드 제품 리스트**:
  - 10개의 쿠키로 바꿀 수 있는 알찬 리워드 카드 6종.
  - 리워드 구성: 
    * 🎟️ `1일 자유 좌석 이용권`
    * 🍱 `급식 1등 하이패스권`
    * 📝 `일일 숙제 면제권`
    * 🎮 `일일 컴퓨터 교실 획득권`
    * 🍿 `모둠 랜덤 과자 상자`
    * 🏷️ `선생님과의 1:1 진로 상담권`
- **의사결정 보증 팝업**:
  - 교환 버튼 클릭 시 가동되는 미니멀한 2단 모달창.
  - "⚠️ 쿠키 10개를 정말로 사용하실 건가요?"
  - 버튼: `[ 확인 (교환 요청) ]` (녹색/파랑) 및 `[ 취소 (돌아가기) ]` (적색/회색).
- **교환 대기 상태 (Pending UI)**:
  - 교환 완료 전에는 추가 요청 제한 및 "선생님께 결재 요청 중..."을 나타내는 스피너 활성화.
- **실시간 차감 완료 세레머니**:
  - 선생님이 승인 시, "교환 수락 완료!" 토스트 알람 발생 및 CSS 쿠키 파티클 이펙트 발생.

#### 3.2 선생님용 (다했니) 기능
- **1반 ~ 10반 반별 그리드 뷰**:
  - 상단에 `1반`, `2반`, ... `10반` 및 `전체보기` 탭 배치.
  - 다했니 오리지널과 유사한 화이트/크림톤 친근한 보드 배치.
- **반별/학생별 오렌지 경고 변색**:
  - 교환 요청이 들어온 반 탭 우측 상단에 강렬한 번쩍이는 알림 점 표시.
  - 해당 교환 요청자 학생의 카드 배경을 `bg-amber-100 border-amber-400 border-2` 등으로 그라데이션 및 골드 펄 효과 적용.
  - 요청 리워드 이름과 요청 시간이 실시간 타임 스태프로 기입되어 한눈에 식별 가능.
- **교환 완료 원 클릭 액션**:
  - `[상점 교환 완료]` 승인 버튼 클릭 시, 즉시 API가 전송되어 쿠키 10개가 즉각 오프라인 차감 리포트 갱신.
  - `[요청 반려]` 버튼으로 혹여나 취소된 경우 쿠키 손상 없이 대기열에서 제외.
- **모의 시뮬레이터 툴바**:
  - 학생이 부족한 8~9개 쿠키일 때 즉시 `+1`을 해줄 수 있도록 교사용 카드 내 미니 `[+1]` 및 `[-1]` 조정 기구 부착.

### 4. API 아키텍처 및 데이터 스키마 (Technical Specs)

#### 데이터 스키마 (In-Memory Database structure)
```typescript
interface Student {
  id: string;          // 고유 식별 번호
  name: string;        // 학생 이름 (예: 김민준, 이서연 등 한국식 친밀한 이름)
  classNo: number;      // 1반부터 10반 (1 - 10)
  code: string;        // 4자리 다했어요 고유 비밀코드 (예: "1001", "1002")
  cookies: number;     // 현재 소지한 쿠키 개수 (초기값 5~15개 랜덤 세팅)
  activeExchange?: {
    itemId: string;    // 요청 상품 ID
    itemName: string;  // 상품 명칭
    requestTime: string; // 요청 시간
  } | null;
}

interface ShopItem {
  id: string;
  name: string;
  cost: number;        // 항상 10
  icon: string;        // 이모지 또는 아이콘
  description: string;
}
```

#### API 끝점 설계
1. `GET /api/state` : 현재 1~10반 모든 학생들의 실시간 쿠키 정보 및 교환 신청 상태 전체 JSON 데이터 리턴.
2. `POST /api/student/login` : `{ code: "1001" }` 방식으로 학생 정보 인증.
3. `POST /api/student/exchange` : `{ studentId: "X", itemId: "Y" }`를 통한 2차 검증 후 교환 신청 등록. (쿠키 10개 보유 여부 검사)
4. `POST /api/teacher/approve` : `{ studentId: "X" }` 교환 승인 처리. 서버 내 해당 학생의 10개 쿠키 강제 차감 및 교환 목록 초기화.
5. `POST /api/teacher/reject` : `{ studentId: "X" }` 교환 요청 반려 처리.
6. `POST /api/teacher/adjust` : `{ studentId: "X", delta: 1 | -1 }` 해당 학생 쿠키 임의 변동 처리 (테스트용 편의성 끝점).
7. `POST /api/state/reset` : 데모 데이터를 리프레시하여 초기 쿠키량으로 리셋.

### 5. UI/UX 및 테마 디자인 가이드라인 (Design Spec)

- **디자인 톤앤매너**:
  - **주 색상(Primary)**: `#FFD23F` (옐로 테마), `#4ECDC4` (민트/민트그린)
  - **보조 색상(Accent)**: `#FF6B6B` (러블리 레드), `#FF9F1C` (오렌지 골드)
  - **배경톤**: `#F9F9FB` (따스한 아이보리 소프트 그레이)
- **모션 피드백**:
  - 버튼 클릭 시 미립자 스케일 축소 효과 (`whileTap={{ scale: 0.95 }}`).
  - 모달 팝업 등장 시 페이드인 및 스프링 바운스 모션 (`motion` 모듈 사용).
  - 요청 수신 시 선생님 보드의 해당 학생 카드가 은은하게 노랗고 오렌지 톤으로 맥박 뛰듯 페이드 펄스 (`animate-pulse`).
- **폴링 주석**:
  - 프론트엔드는 2초 주기로 백엔드 `/api/state`를 자동 API 동기화하여 별도의 복잡한 웹소켓 셋업 부담 없이 초고속 실시간성을 보장하고 부하를 최소화합니다.
